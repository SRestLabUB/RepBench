# LLM-Based Vulnerability Detection Pipeline

A complete end-to-end pipeline for detecting software vulnerabilities using Joern code analysis and LLM-based reasoning.

## Overview

This project implements an automated vulnerability detection pipeline:
- **Joern Analysis**: Generate AST, CFG, PDG representations
- **DOT Conversion**: Parse and linearize graph representations
- **Conservative Compression**: Reduce prompt size while preserving evidence
- **Dynamic Few-Shot**: Select examples based on prompt budget
- **Chain-of-Thought**: 5-step reasoning process for LLM
- **LLM API Client**: Environment-configured OpenAI-compatible endpoint

## Supported CWEs

- **CWE-121**: Stack-based Buffer Overflow
- **CWE-122**: Heap-based Buffer Overflow
- **CWE-190**: Integer Overflow
- **CWE-191**: Integer Underflow
- **CWE-415**: Double Free
- **CWE-416**: Use After Free

## PrimeVul Testcase Generation Notes

PrimeVul `file_contents/` is raw full-file data. It should not be scanned directly as the testcase list. Testcases are generated by `bin/extract_primevul_testcases.py`, which iterates through `primevul_test_paired.jsonl`, `primevul_train_paired.jsonl`, and `primevul_valid_paired.jsonl`, then uses `file_info.json` to map each `func_hash` to the raw file in `file_contents/`.

Generation command:

```bash
python3 bin/extract_primevul_testcases.py \
  -i Joern_llm_implement/primevul_data \
  -o primevul_testcases_output_clean
```

Generated layout:

```text
primevul_testcases_output_clean/
└── <project>/
    ├── project_src/                         # cloned once per project
    └── <CWE>/
        └── <file_name>/
            └── <commit_id>/
                ├── cse713_fixed/
                │   ├── <function_name>      # copied source file/function context from file_contents
                │   ├── metadata.json        # CWE/CVE/target/project_file_path/function lines
                │   └── project_src/         # repo checkout at fixing commit
                └── cse713_vulnerable/
                    ├── <function_name>
                    ├── metadata.json
                    └── project_src/         # repo checkout at commit_id~1
```

Observed clean generation result on the current local dataset:

- Generated testcase metadata files: 107
- Unique commit testcase groups: 55
- Groups with both `cse713_fixed` and `cse713_vulnerable`: 52
- Groups missing one side: 3
- Missing/invalid per-testcase `project_src/.git`: 0
- Missing copied source/function file: 0
- CWE distribution: CWE-190(56), CWE-415(23), CWE-416(22), CWE-122(4), CWE-191(2)

Important caveats:

- Do not infer final dataset size only from JSONL pre-counts. The script skips records whose `func_hash` is absent from `file_info.json`; this accounted for 19 skipped records in the current local run.
- Existing output directories may contain old `fixed/` and `vulnerable/` labels from previous script versions. New pipeline code should consume only `cse713_fixed/` and `cse713_vulnerable/`.
- Re-running into a partially generated output directory can leave mixed old/new labels. Prefer a fresh output directory for reproducible statistics.
- Downstream Joern validation should compare both inputs for 1-2 sample pairs first: the copied function/file snippet in each label directory and the full checked-out file under `project_src/<project_file_path>` from `metadata.json`.

Initial shapelib CWE-415 comparison result:

- The copied `split` file in `cse713_fixed/` and `cse713_vulnerable/` is named after the function, but its contents are the full raw source file copied from `file_contents/`, not a function-only snippet.
- A true snippet can be extracted from `project_src/<project_file_path>` using `metadata.function_start` and `metadata.function_end`.
- Joern method-filtered queries (`cpg.method.name("split")`) produced identical target-function graph sizes for true snippet input and full-file input.
- Full-file all-method queries were much larger and introduced prompt-budget pressure. In the shapelib pair, the all-method vulnerable prompt was misclassified as SAFE, while both method-filtered modes were classified correctly.
- Recommended PrimeVul mode for the main pipeline: import the checked-out full file for parse correctness, but export/filter only the target function graph by function name before building the LLM prompt.

## Quick Start

### 1. Install Dependencies

```bash
python -m pip install -r ../requirements.txt
```

### 2. Configure Runtime Settings

Normally, do **not** set `CSE713_ROOT`. `project_paths.py` automatically finds the checkout directory by locating a parent containing both `README.md` and `Joern_llm_implement/`. Application scripts import paths from it; users do not run it directly:

```python
from project_paths import PROJECT_ROOT
output_dir = PROJECT_ROOT / "outputs"
```

For online runs, set `LLM_API_KEY` and `LLM_API_BASE_URL` (including `/v1`) in the environment. In PowerShell, set the same variables through the environment-variable syntax. Optional overrides are `CSE713_ROOT`, `CSE713_JULIET_BASE`, `CSE713_REPRESENTATIONS_BASE`, and `JOERN_JAVA_HOME`; set them only when automatic discovery does not apply.

### 3. Install Joern
find the joern-install.sh file in the repo

```
chmod +x ./joern-install.sh
sudo ./joern-install.sh
joern
```

by doing so, you can see the version of the Joern, type "exit" to exit

### 4. Run Demo

```bash
python3 demo_llm_pipeline.py --dry-run  # no API call
python3 demo_llm_pipeline.py            # online run
```

### 5. Run Batch Validation

```bash
python3 validate_llm_samples.py --limit 3 --model qwen
```

## Standard Pipeline

1. Generate AST, CFG, and PDG with `joern_http_client.py`.
2. Convert or compress DOT graphs with `dot_converter.py` and `dot_compressor.py`.
3. Build a budget-aware prompt with source, graph evidence, and selected few-shot examples.
4. Use `demo_llm_pipeline.py --dry-run` to inspect the prompt, or run without `--dry-run` to call the configured LLM.
5. Use `validate_llm_samples.py` for batch evaluation.

Typical representation commands:

```bash
python3 joern_http_client.py --cwe CWE-190 --file test_case_01.c
python3 joern_http_client.py --cwe CWE-190 --batch --limit 10
python3 dot_converter.py file.pdg.dot --format text
```

## Key Features

### Dynamic Few-Shot Selection
- Selects examples based on prompt budget
- Drops examples if budget insufficient
- Prioritizes core representations

### Conservative DOT Compression
- Preserves IDENTIFIER and LITERAL nodes
- Drops BLOCK nodes without CWE evidence
- 10-30% reduction target

### Chain-of-Thought Reasoning
5-step process:
1. Identify Sink (dangerous operations)
2. Trace Source (data origins)
3. Check Validation (bounds/null checks)
4. Analyze Flow (CFG control flow)
5. Determine (VULNERABLE or SAFE)
